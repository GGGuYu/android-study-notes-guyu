# 项目性能优化笔记 - 面试专用

## 1. Player复用池 - 短视频场景优化

### 项目场景
我们的音乐App有短视频播放功能（类似抖音），用户可以快速上下滑动切换视频。初期实现是每次滑动都创建新的`ExoPlayer`实例，导致**内存频繁分配/回收**，用户滑动多了之后App开始卡顿，甚至出现OOM。

### 优化方案
我们实现了一个**Player复用池**，预创建5个`ExoPlayer`实例，通过取模运算实现复用：

```kotlin
// 预创建5个Player
val players = listOf(
    createPlayer(), createPlayer(), createPlayer(),
    createPlayer(), createPlayer()
)

// 滑动时复用
fun onPageChange(index: Int) {
    players[currentIndex % 5].pause()  // 暂停当前页
    players[index % 5].play()          // 复用已有Player播放新视频
}
```

### 原理说明
- **对象复用**：避免频繁的`new/destroy`带来的GC压力
- **池化思想**：类似线程池、连接池，控制资源数量上限
- **为什么是5个**：覆盖当前页+上下各2页的预加载范围，平衡内存占用和滑动体验

### 面试话术
> "考虑到短视频场景的快速滑动特性，频繁创建Player会导致内存抖动。我们采用池化设计，复用5个Player实例，就像RecyclerView复用ViewHolder一样。"

## 2. ReplaceableForwardingPlayer - 播放器状态迁移

### 项目场景
用户可以从**短视频页面**点击歌曲进入**全屏音乐播放器**，需要无缝切换播放。如果直接释放短视频Player再创建新Player，会出现**播放中断、进度丢失、音量重置**等问题，体验很差。

### 优化方案
我们实现了`ReplaceableForwardingPlayer`作为代理层：

```kotlin
class ReplaceableForwardingPlayer(private var player: Player) : Player {
    fun setPlayer(player: Player) {
        // 1. 转移监听器
        listeners.forEach { oldPlayer.removeListener(it); newPlayer.addListener(it) }
        // 2. 转移播放状态（进度、音量、循环模式）
        newPlayer.volume = oldPlayer.volume
        newPlayer.setMediaItems(playlist, currentIndex, oldPlayer.contentPosition)
        // 3. 清理旧Player但不释放
        oldPlayer.clearMediaItems()
        oldPlayer.stop()
    }
}
```

### 原理说明
- **代理模式**：对外暴露统一接口，内部可动态替换实现
- **状态迁移**：播放进度、音量、循环模式等状态无损转移
- **延迟释放**：旧Player只是清理资源，回到复用池等待下次使用

### 面试话术
> "这里用到了代理模式。ReplaceableForwardingPlayer作为统一接口，内部可以无缝切换Player实例。当用户从短视频进入全屏播放时，我们转移播放进度、音量等状态，用户感知不到切换过程。"

## 3. VerticalPager预加载 - 流畅滑动体验

### 项目场景
短视频列表使用`VerticalPager`实现，如果不预加载，用户滑到新页面时才开始加载视频，会出现**黑屏或加载中**的等待时间，体验很差。

### 优化方案
设置`beyondBoundsPageCount = 1`，预加载当前页上下各1页：

```kotlin
VerticalPager(
    state = pagerState,
    beyondBoundsPageCount = 1,  // 预加载屏幕外1个页面
) { page ->
    VideoPlayer(content = datum[page])
}
```

### 原理说明
- **空间换时间**：用内存占用换取加载速度
- **懒加载边界**：Compose只组合可见+预加载范围的页面
- **为什么不是2页**：预加载太多会增加内存压力，1页足够应对快速滑动

### 面试话术
> "我们设置了`beyondBoundsPageCount = 1`，让用户滑动时下一页已经预加载好了。这个值需要权衡，太大增加内存压力，太小又会出现白屏。"

## 4. collectAsStateWithLifecycle - 生命周期感知

### 项目场景
使用`collectAsState()`收集Flow时，即使App退到后台（如用户接电话），StateFlow仍然在收集状态，导致**不必要的CPU消耗和UI重组**。

### 优化方案
使用`collectAsStateWithLifecycle()`替代`collectAsState()`：

```kotlin
// ❌ 原来：即使后台也持续收集
val state by viewModel.stateFlow.collectAsState()

// ✅ 优化：只在Lifecycle RESUMED时收集
val state by viewModel.stateFlow.collectAsStateWithLifecycle()
```

### 原理说明
- **Lifecycle绑定**：自动监听Activity/Fragment生命周期
- **非RESUMED状态暂停**：App后台或页面不可见时自动停止收集
- **自动恢复**：页面恢复时自动恢复收集，无需手动管理

### 面试话术
> "原来用collectAsState，即使App在后台也会持续收集，造成资源浪费。换成collectAsStateWithLifecycle后，它会监听生命周期，只有页面在前台时才收集，后台自动暂停。"

## 5. SharingStarted.WhileSubscribed - Flow共享优化

### 项目场景
多个UI组件同时观察同一个StateFlow（如用户信息）。如果没有共享策略，每个订阅者都会触发一次上游数据获取，造成**重复请求**。

### 优化方案
使用`WhileSubscribed`策略：

```kotlin
val userData = repository.userData
    .stateIn(
        scope = coroutineScope,
        initialValue = UserData(),
        started = SharingStarted.WhileSubscribed(5_000)
    )
```

### 原理说明
- **共享机制**：多个订阅者共享同一个Flow，上游只执行一次
- **超时机制**：`5_000ms`表示最后一个订阅者取消后，保持5秒再停止
- **避免重启**：短暂页面跳转（如弹窗）不会导致Flow重新启动

### 面试话术
> "我们用WhileSubscribed设置5秒超时。这样配置变更或短暂跳转时，Flow不会停止，回来还能继续使用缓存的数据。如果不用这个，每次配置变更都会重新请求。"

## 6. snapshotFlow - 滚动监听优化

### 项目场景
短视频滑动时需要知道当前页码，但Compose的`PagerState`是State对象，如果用`remember`直接观察会导致**频繁重组**。

### 优化方案
使用`snapshotFlow`将Compose状态转为Flow：

```kotlin
// 避免频繁重组，只在页码变化时触发
snapshotFlow { pagerState.settledPage }.collect { pageIndex ->
    viewModel.onPageChange(pageIndex)
}
```

### 原理说明
- **状态隔离**：snapshotFlow将Compose状态从组合中解耦
- **精准触发**：只在值真正变化时收集，不会因为State对象引用变化而触发
- **性能优势**：避免Composable不必要的重组

### 面试话术
> "Compose状态变化会导致重组，但有时候我们只关心值的变化。snapshotFlow把Compose状态转成Flow，只有值变化时才回调，不会引起重组。"

## 7. Room批量操作 - 数据库性能

### 项目场景
歌单详情页需要展示100+首歌曲，初期用循环逐个插入数据库，导致**UI卡顿**，插入100条数据需要2-3秒。

### 优化方案
使用Room的批量插入和ConflictStrategy：

```kotlin
@Dao
interface SongDao {
    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insertAll(entities: List<SongEntity>)  // 批量插入
}

// 使用
songDao.insertAll(songs)  // 一次性插入，底层是事务
```

### 原理说明
- **事务优化**：批量操作在一个事务中执行，减少IO次数
- **REPLACE策略**：实现UPSERT语义，插入或更新一行代码搞定
- **Flow响应式**：查询使用Flow，数据变化自动刷新UI

### 面试话术
> "批量操作比循环插入快很多，因为它在一个事务里完成。我们还用了REPLACE策略，自动处理插入或更新，代码更简洁。"

## 8. 进度更新频率控制 - 播放体验

### 项目场景
播放进度需要实时更新进度条，初期用`Handler.postDelayed(100ms)`，发现**CPU占用高**，且快速滑动时进度跳动明显。

### 优化方案
优化为16ms间隔（约60fps），并加入缓存机制：

```kotlin
publishProgressJob = scope.launch {
    while (playbackState.value.isPlaying) {
        currentPosition.value = mediaController?.currentPosition ?: 0
        delay(16.milliseconds)  // 60fps，视觉流畅
        
        // 每2秒保存一次播放进度到数据库
        if (currentPosition.value - lastPosition >= 2000) {
            userDataRepository.saveRecentSong(...)
            lastPosition = currentPosition.value
        }
    }
}
```

### 原理说明
- **帧率匹配**：16ms ≈ 60fps，与屏幕刷新率匹配，视觉最流畅
- **频率降级**：数据库保存降至2秒一次，减少IO压力
- **协程优势**：`delay`不会阻塞线程，比Handler更高效

### 面试话术
> "我们用16毫秒间隔更新进度条，对应60fps。但保存到数据库太频繁会有IO压力，所以改成2秒存一次。用协程delay比Handler更高效，不会阻塞线程。"

## 9. 歌词事件背压处理 - 防止内存溢出

### 项目场景
歌词逐字高亮需要发送时间同步事件，如果用户快速拖动进度条，会产生大量事件，可能导致**内存溢出**。

### 优化方案
配置事件总线缓冲区策略：

```kotlin
@EmoEventProp(
    sticky = false,
    keepChannelAlive = false,
    extraBufferCapacity = 1,  // 缓冲区只存1个事件
    onBufferOverflow = BufferOverflow.DROP_OLDEST  // 满了丢弃旧的
)
class TipEvent(...)
```

### 原理说明
- **背压策略**：缓冲区满了丢弃旧事件，保证最新事件处理
- **容量控制**：`extraBufferCapacity = 1`限制内存占用
- **Channel复用**：`keepChannelAlive = false`不保持Channel，节省资源

### 面试话术
> "歌词事件如果积压会占内存。我们设置了缓冲区大小为1，满了就丢弃旧的，保证处理最新的。这样即使有事件风暴也不会OOM。"

## 10. DataStore Protobuf序列化 - 高效存储

### 项目场景
应用配置（播放模式、主题设置、用户偏好）需要持久化，初期用SharedPreferences，但**读写性能差**，且不支持异步监听。

### 优化方案
使用DataStore + Protobuf：

```kotlin
// 定义Proto schema
syntax = "proto3";
message UserDataPreferences {
  int32 playback_mode = 1;
  bool dark_theme = 2;
  string user_id = 3;
}

// 使用DataStore
DataStoreFactory.create(
    serializer = protobufSerializer,
    scope = CoroutineScope(Dispatchers.IO)  // IO线程执行
) { context.dataStoreFile("user_preferences.pb") }
```

### 原理说明
- **Protobuf优势**：二进制格式，比XML/JSON体积小、解析快
- **类型安全**：编译期检查，避免SharedPreferences的getXXX类型错误
- **协程支持**：天然支持Flow，数据变化自动通知
- **事务安全**：保证数据一致性，不会读写冲突

### 面试话术
> "我们用DataStore替代SharedPreferences，配合Protobuf序列化。Protobuf是二进制格式，比XML快很多。而且支持Flow监听，配置变化自动刷新UI，不需要手动通知。"

## 11. OkHttp TCP连接池优化 - 音视频下载性能

### 项目场景
我们的音乐App涉及大量的音视频下载、封面图加载、歌词文件请求等HTTP操作。初期使用OkHttp默认配置，但在**弱网环境**或**快速切换歌曲**时，频繁出现"等待连接"的延迟，用户体验不流畅。通过日志分析发现，默认连接池参数对音视频这种**高并发、长连接**场景不够友好。

### 优化方案
针对音视频项目特性，调整`ConnectionPool`参数：

```kotlin
@Provides
@Singleton
fun providesOkHttpClient(): OkHttpClient {
    return OkHttpClient.Builder()
        // 默认配置：maxIdleConnections=5, keepAliveDuration=5min
        // 优化配置：提高并发连接数和保活时间
        .connectionPool(
            ConnectionPool(
                maxIdleConnections = 10,      // 从5提升到10
                keepAliveDuration = 10,       // 从5分钟提升到10分钟  
                timeUnit = TimeUnit.MINUTES
            )
        )
        // 复用HTTP/2连接（OkHttp默认支持，确保服务端也开启）
        .protocols(listOf(Protocol.HTTP_2, Protocol.HTTP_1_1))
        
        // 连接超时适当延长，弱网环境更友好
        .connectTimeout(15, TimeUnit.SECONDS)  // 从10秒延长到15秒
        .readTimeout(30, TimeUnit.SECONDS)     // 音视频下载需要更长的读取时间
        .writeTimeout(10, TimeUnit.SECONDS)
        .build()
}
```

### 原理说明

**ConnectionPool核心参数：**
- **`maxIdleConnections`**：连接池中保持空闲的最大连接数
  - 默认5个，音视频项目并发下载多（同时下载歌曲、封面、歌词），容易打满
  - 提升到10个，支持更多并发连接复用

- **`keepAliveDuration`**：连接保活时间
  - 默认5分钟，用户听歌间隔可能超过5分钟，导致连接失效需要重新建立TCP三次握手
  - 提升到10分钟，减少重复建连开销

**HTTP/2多路复用：**
- OkHttp默认支持HTTP/2，一个TCP连接可以承载多个请求流
- 确保服务端开启HTTP/2，才能真正实现连接复用

**超时策略：**
- 音视频文件较大，`readTimeout`需要比API请求更长
- `connectTimeout`适当延长，适应弱网环境的TCP建连

### 实际收益
- **连接复用率**：从约60%提升到85%+
- **首包时间**：平均减少150-300ms（避免了TCP三次握手）
- **弱网体验**：连接超时导致的失败率降低40%

### 面试话术
> "音视频项目特点是并发下载多、文件大、连接持续时间长。我们调大了ConnectionPool的maxIdleConnections到10，keepAliveDuration到10分钟，这样用户在切歌时，TCP连接还在池子里可以直接复用，避免了重复的三次握手。同时把readTimeout延长到30秒，适应大文件下载。"

## 12. LaunchedEffect线程切换 - 避免主线程阻塞

### 项目场景
刚开始用Jetpack Compose时，我以为`LaunchedEffect`会像React的`useEffect`一样自动在后台执行异步操作。于是我在里面直接调用了网络请求：

```kotlin
// ❌ 错误写法：在主线程执行网络请求，导致ANR
LaunchedEffect(Unit) {
    val result = api.fetchData()  // 阻塞主线程！
    data.value = result
}
```

结果页面加载时明显卡顿，甚至出现ANR（Application Not Responding）。通过Profiler分析发现，网络请求在主线程执行，阻塞了UI渲染。

### 优化方案
理解了Compose的机制后，我们将网络请求切换到IO线程执行：

```kotlin
// Repository层：使用flowOn(Dispatchers.IO)切换线程
class FeedRepository @Inject constructor(
    private val network: MyNetworkDatasource,
) {
    fun feeds(): Flow<NetworkResponse<...>> = flow {
        emit(network.feeds())
    }.flowOn(Dispatchers.IO)  // ✅ 网络请求在IO线程
}

// ViewModel层：使用viewModelScope（默认Main）
class FeedViewModel : BaseViewModel() {
    fun loadData() {
        viewModelScope.launch {
            repository.feeds()
                .collect { result ->
                    // 数据回调在主线程，更新UI
                    datum.value = result
                }
        }
    }
}

// UI层：安全地调用
LaunchedEffect(Unit) {
    viewModel.loadData()  // 异步触发，不阻塞
}
```

### 原理说明

**LaunchedEffect的执行线程**：
- `LaunchedEffect`默认在`Dispatchers.Main`（主线程）启动协程
- 适合轻量级操作（如状态监听、简单的UI计算）
- **不适合**耗时操作（网络请求、数据库查询、文件IO）

**协程线程切换策略**：
- **`flowOn(Dispatchers.IO)`**：上游操作（网络请求）在IO线程，下游收集自动切回主线程
- **`withContext(Dispatchers.IO)`**：在协程内部临时切换到IO线程
- **`viewModelScope.launch(Dispatchers.IO)`**：整个协程在IO线程运行

**为什么不用`rememberCoroutineScope`**：
- `rememberCoroutineScope`创建的Scope也是主线程
- 需要手动指定`Dispatchers.IO`，容易遗漏

### 最佳实践总结

```kotlin
// ✅ 推荐：Repository统一处理线程切换
class Repository {
    fun fetchData() = flow {
        emit(api.getData())
    }.flowOn(Dispatchers.IO)
}

// ✅ LaunchedEffect只负责触发，不处理耗时逻辑
LaunchedEffect(Unit) {
    viewModel.loadData()  // 轻量级调用
}
```

### 面试话术
> "刚开始用Compose时我犯了一个错误，以为LaunchedEffect像React的useEffect一样是异步的。后来发现它默认在主线程执行，如果在里面直接进行网络请求会阻塞UI。我们现在的做法是在Repository层统一使用`flowOn(Dispatchers.IO)`，这样UI层只需要触发请求，数据会自动回调到主线程更新UI，不用担心线程问题。"

## 面试金句总结

1. **对象复用**："池化设计避免频繁创建销毁，就像线程池、连接池一样"
2. **状态迁移**："用代理模式无缝切换Player，用户感知不到切换过程"
3. **生命周期**："collectAsStateWithLifecycle自动管理生命周期，后台不浪费资源"
4. **批量操作**："事务批量插入比循环快一个数量级"
5. **频率控制**："16ms对应60fps，既流畅又不浪费CPU"
6. **背压处理**："丢弃旧事件保证最新，防止事件风暴OOM"
7. **存储优化**："Protobuf比XML/JSON更小更快，而且类型安全"
8. **连接复用**："调大ConnectionPool参数，切歌时直接复用TCP连接，避免三次握手"
9. **线程安全**："LaunchedEffect默认主线程，耗时操作必须切到IO线程，避免ANR"

## 可补充的优化方向（如果被追问）

### 关于DNS优化
> "如果面试官问到DNS优化，我们可以说可以考虑接入HTTPDNS，避免运营商DNS劫持和解析慢的问题，特别是音视频项目对首包时间敏感。"

### 关于图片加载
> "我们用Coil加载封面图，它内置了内存和磁盘缓存。对于列表中的图片，还可以考虑设置placeholder和error图片，避免布局抖动。还可以开启图片渐进式加载（渐进式JPEG）提升感知速度。"

### 关于内存优化
> "ExoPlayer有`CacheDataSource`可以缓存音视频数据，重复播放时直接从缓存读，减少网络请求。还可以使用`SimpleCache`配合`LeastRecentlyUsedCacheEvictor`实现LRU缓存策略。"

### 关于APK体积
> "目前项目`isMinifyEnabled = false`，发布时可以开启R8代码压缩和ProGuard混淆，配合资源压缩`shrinkResources`，能显著减小APK体积。"
